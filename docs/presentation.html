01The goal

Build a text-to-speech system, fine-tune it, and evaluate it. Graded on the ML process & analysis — not on beating state of the art.

Input

English text, e.g. "Roger roger."

Model

VITS — one network, text → audio waveform.

Twist

Make it speak in the droid voice, learned natively.

02How it works: fine-tuning

A TTS model learns from thousands of (text, audio) pairs and speaks in whatever voice those recordings use. We don't train from scratch — we fine-tune a strong pretrained checkpoint (kakao-enterprise/vits-ljs) and adapt it. Still real training: real loss curve, steps, GPU-hours. (Assignment Option C: VITS, end-to-end.)

03The idea — voice distillation

No real droid recordings exist, so we manufacture the target voice: an RVC converter acts as a teacher that recolours speech into the droid timbre. We bake a droid dataset once, then train VITS to reproduce that voice natively — the teacher isn't needed at run time.

[1] sourceLJSpeech13,100 (text + natural voice)

Why distillation, not "TTS → RVC at inference": this makes the droid the model's own voice → a real train/measure/report experiment, not just glued tools.

04The data

Source — LJSpeech

~13,100 clips, single voice. Last 500 held out as the test set (never trained on).

Target — ljspeech-b1

Dmi1tr13/ljspeech-b1 on the HF Hub. 13,100 droid-voiced clips, 22050 Hz. Same transcripts — only the timbre changed.

05Training

Fine-tuned VITS on Colab (T4) — batch 2, LR 2e-4, on a 4,000-clip subset, to 2,000 steps (matching the baseline setting).

Loss dropped fast for ~50 steps, then flat-lined. That flat KL is the warning sign (see slide 07).

06Results — measured with Whisper

We didn't judge by ear — we ran Whisper (ASR) on the output and computed WER: low = intelligible, ~100% = noise. An objective, reproducible check.

ModelWER ↓Whisper heard…
Pretrained VITS10–20%"roger roger, all units, proceed with the mission, standing by" ✅
Fine-tuned · 1000 steps90%"boderaer or i'll get it. with magic, hey, it goes!" ❌
Fine-tuned · 2000 steps120%"sorry, i owe ya everfield if thy he monday get spien i" ❌

Honest negative result: the fine-tune produces noise at both 1000 and 2000 steps — while the same code on the pretrained checkpoint is fully intelligible. More steps didn't help.

07Why it didn't work (the lesson)

Low training loss ≠ good audio.

VITS trains on one path (reconstruct audio from the posterior of real audio) but generates on another (prior + duration predictor + decoder). Our recon loss looked okay, but KL never converged (~68) — so the generation path never learned to match the decoder. At inference it feeds the decoder out-of-distribution latents → noise.

Most likely root cause: the objective is reconstruction + KL + duration only — no adversarial (discriminator) loss, which real VITS relies on to train the decoder. That's why 2000 steps ≈ 1000 steps: more of an incomplete objective doesn't fix it.

Next step: add the HiFi-GAN-style discriminator/adversarial loss (or use an established VITS fine-tune recipe); and verify what the repo's droid_test.wav was actually produced by.

08Engineering challenges

  • Reproducible env on Colab: the VITS tokenizer needs the system package espeak-ng; the pinned torch+cu128 (a local GPU build) had to be swapped for Colab's torch.
  • Test-set leakage — caught & fixed: the training flow loaded the whole dataset incl. the 500 held-out test texts → now always held out.
  • espeak memory-map leak: phonemizer re-loads the espeak C lib every call and never frees it → the run hangs after ~1000 steps. Worked around with a subset + workers + resume. (A crash that got worse with fewer processes → a per-process leak, not a concurrency bug.)
  • Objective evaluation: used Whisper WER instead of subjective listening — which is exactly what exposed the negative result.
